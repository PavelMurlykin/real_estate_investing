# Ролевая модель приложения

Документ описывает прикладные роли `real_estate_investing`, их права и правила,
которые нужно сохранять при доработке backend, templates и тестов.

## Источник прав

Основной код ролевой модели находится в `users/roles.py`.

Роли назначаются через стандартные группы Django:

- `moderators` - модераторы приложения.
- `application_administrators` - администраторы приложения.

Группы создаются миграцией `users/migrations/0002_create_role_groups.py`.
Отдельной страницы управления ролями в приложении нет: назначение пользователей
модераторами или администраторами выполняется через Django admin.

`is_superuser=True` не является отдельной прикладной группой, но в коде
приравнивается к администратору приложения.

## Иерархия ролей

Роли расширяют права нижележащих ролей:

1. Неавторизованный пользователь.
2. Авторизованный пользователь.
3. Модератор.
4. Администратор приложения.
5. Django superuser.

## Правила владения данными

Персональные данные и сохраненные расчеты должны иметь владельца:

- `Customer.user`
- `MortgageCalculation.user`
- `TrenchMortgageCalculation.user`

Обычный авторизованный пользователь, включая модератора, может читать,
изменять, удалять и связывать только объекты, где владелец совпадает с
`request.user`.

Администратор приложения и superuser могут работать со всеми клиентами и
сохраненными расчетами.

Объекты без владельца не должны быть доступны обычным пользователям. Такие
записи допустимы только как исторические данные или техническое состояние после
миграций; доступ к ним должен оставаться на уровне администратора приложения.

## Backend-контракт

Проверки доступа должны выполняться на backend, а не только через скрытие
кнопок в templates.

Использовать helpers из `users/roles.py`:

- `can_view_private_records(user)` - доступ к разделам клиентов и сохраненных
  расчетов для авторизованных пользователей.
- `can_view_all_private_records(user)` - доступ ко всем клиентам и расчетам для
  администратора приложения или superuser.
- `can_manage_catalogs(user)` - право менять глобальные справочники `bank`,
  `location`, `property`.
- `can_sync_external_data(user)` - право запускать синхронизацию внешних данных.
- `CatalogManagementRequiredMixin` - защита class-based views для CRUD
  справочников.
- `ExternalDataSyncRequiredMixin` - защита class-based views для sync-действий.

Для всех view, которые получают `customer_id`, `calculation_id` или похожий ID
из URL, формы или POST, queryset должен быть предварительно ограничен правами
текущего пользователя. Нельзя сначала получить объект по ID, а потом проверять
его владельца только в template.

## Mortgage-контракт

Неавторизованный пользователь может открыть `/mortgage/`, выполнить расчет и
выгрузить результат в Word или Excel, но такой сценарий не должен создавать
записи в базе данных.

Авторизованный пользователь может сохранять расчеты. Владелец сохраненного
расчета определяется так:

- если расчет сохраняется для выбранного клиента, владелец расчета должен
  совпадать с владельцем клиента;
- если клиент не выбран, владелец расчета - `request.user`.

Создание глобальных объектов недвижимости из формы расчета допустимо только для
пользователей с `can_manage_catalogs=True`. Обычный пользователь может считать
по ручным данным, но не должен через калькулятор создавать новые глобальные
объекты `property`.

## Template-контракт

Флаги ролей доступны во всех templates через context processor
`users.context_processors.application_roles`:

- `is_moderator`
- `is_application_administrator`
- `can_manage_catalogs`
- `can_sync_external_data`
- `can_view_private_records`
- `can_view_all_private_records`

Templates должны скрывать недоступные действия, но это считается только
улучшением интерфейса. Безопасность должна обеспечиваться view/queryset/form
логикой на backend.

## Контракт для тестов

При изменении ролевой модели или связанных view нужно сохранять покрытие для
следующих сценариев:

- anonymous может читать публичные справочники и не видит mutation-действий;
- anonymous может считать ипотеку и экспортировать результат без сохранения в
  базу;
- anonymous не может открыть клиентов и сохраненные расчеты;
- обычный пользователь видит только своих клиентов и свои расчеты;
- обычный пользователь не может получить чужие объекты по угаданному ID;
- модератор может выполнять CRUD справочников, но не может запускать sync;
- администратор приложения может запускать sync и видеть все частные записи;
- superuser получает права администратора приложения.

## Чеклист для будущих изменений

Перед добавлением нового endpoint, формы или кнопки проверить:

1. Это публичные данные или данные конкретного пользователя?
2. Нужен ли для записи `can_manage_catalogs`?
3. Нужен ли для sync/import/export с внешним источником `can_sync_external_data`?
4. Должен ли queryset фильтроваться по `user=request.user`?
5. Должен ли администратор приложения видеть все записи?
6. Не сохраняет ли anonymous-сценарий данные в БД?
7. Скрыты ли недоступные действия в template?
8. Есть ли тест на запрет доступа по прямому URL или угаданному ID?
